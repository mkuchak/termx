# F-Droid submission notes

This doc captures the mechanics of getting termx onto
[F-Droid](https://f-droid.org/) and what to check before each release.

## Layout

- `fastlane/metadata/android/en-US/` — user-visible listing content.
  Mirrors Triple-T/gradle-play-publisher's layout so the same tree feeds
  F-Droid and (some day) Play Console.
  - `title.txt` — short app name ("termx").
  - `short_description.txt` — single-sentence pitch.
  - `full_description.txt` — two-paragraph listing, <4k characters.
  - `changelogs/<versionCode>.txt` — one file per release; the bot reads
    the file whose name matches the APK's `versionCode`. The versionCode
    is derived monotonically from `git rev-list --count HEAD` in
    `android/app/build.gradle.kts`.
  - `images/icon.png` — 512x512 PNG icon placeholder. **Replace before
    submitting.**
  - `images/phoneScreenshots/1..5.png` — transparent placeholders
    generated by Task #49. **Replace before submitting.**
- `metadata/dev.kuch.termx.yml` — the server-side manifest the F-Droid
  build bot actually consumes. Categories, license, source URL, build
  recipe, and the auto-update regex live here.

## Submission workflow

1. Cut a release (`npm run release` → tags `vX.Y.Z`).
2. Fork https://gitlab.com/fdroid/fdroiddata, drop the YAML into
   `metadata/dev.kuch.termx.yml`, and open a merge request. The initial
   MR also ships the listing (`fastlane/metadata/`) — after first
   publish, listing changes auto-flow from our repo via the `Repo:`
   link.
3. The fdroiddata CI runs the build recipe on a clean VM against the
   tagged commit and verifies the APK installs. If the build
   "reproduces" (hash-matches an APK signed by the fdroid server), the
   release is promoted.

## Reproducible builds

**Important:** F-Droid does *not* run our CI or use our keystore. It
checks out the source at the commit referenced in the manifest and
rebuilds the APK from scratch with a fresh JDK/AGP toolchain, then
signs with the F-Droid keystore. Our CI's signing step is irrelevant to
reproducibility on the F-Droid side; what matters is that the same
source produces the same bytes when built twice.

Known non-deterministic sources we already handle:

- `versionCode` derived from `git rev-list --count HEAD` is stable for a
  given commit.
- AGP 8.12 + R8 `isMinifyEnabled = true` in `app/build.gradle.kts`
  produces stable output for identical inputs provided the Gradle
  wrapper version is pinned. The wrapper is checked in
  (`android/gradle/wrapper/gradle-wrapper.properties`).
- The only NDK binary we ship is `libmosh_client.so` in
  `jniLibs/`, pre-compiled per ABI. F-Droid's reproducible-build
  policy allows pre-built binaries only when they are themselves
  reproducibly rebuildable from source — the mosh upstream is, and we
  lock to a specific release tag in the build scripts.

To verify locally that two consecutive builds match byte-for-byte:

```bash
cd android
./gradlew :app:assembleRelease
sha256sum app/build/outputs/apk/release/termx-v*.apk
./gradlew :app:clean :app:assembleRelease
sha256sum app/build/outputs/apk/release/termx-v*.apk
```

The two `sha256sum` outputs should match. If they don't:

- Check `APK_SIGNATURE_V2` differences — the F-Droid server re-signs
  anyway, so `apksigner verify --print-certs` on the two APKs isn't a
  mismatch source per se, but `diffoscope` output on the unsigned APK
  will show the real non-determinism.
- Make sure `BUILD_TIMESTAMP` / `SOURCE_DATE_EPOCH` are not being
  embedded (our `BuildConfig` fields don't reference dates, confirmed
  by eyeballing `app/build.gradle.kts`).

## Pre-submission checklist

- [ ] Bump version via `npm run release` and push the tag.
- [ ] Wait for `android-release.yml` CI to produce the release APK.
- [ ] Verify the `versionCode` of the released APK matches the number
  in `fastlane/metadata/android/en-US/changelogs/<versionCode>.txt`.
- [ ] Replace placeholder PNGs in `fastlane/metadata/android/en-US/images/`
  with real screenshots and icon.
- [ ] Update `CurrentVersion` + `CurrentVersionCode` in
  `metadata/dev.kuch.termx.yml` to match the new tag.
- [ ] Open the fdroiddata MR.
